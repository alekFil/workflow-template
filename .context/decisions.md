## ADR-014: Публикация как OSS — перевод, слэш-команды, демо

**Статус:** Принято

**Контекст:** Репо достигло состояния, пригодного для публикации как OSS-инструмента для сообщества CC-пользователей. Продукт — workflow-слой (скиллы, структура `.context/`, install/uninstall), не project starter с кодовым скаффолдингом.

**Решение:**

1. **Перевод на английский** — все файлы, которые видит пользователь: `template/CLAUDE.md`, `template/WORKFLOW.md`, `template/.claude/`, `template/.context/`, `README.md`, `SETUP.md`, `scripts/install.sh`, `scripts/uninstall.sh`.

2. **Слэш-команды вместо ключевых фраз** — естественные фразы (`обсудим`, `начинаем реализацию` и др.) ненадёжны: срабатывают случайно, непредсказуемы для нового пользователя. Заменить на детерминированные `/команды` через механизм CC slash commands.

3. **Демо** — GIF или скринкаст: одна сессия от `/architect` до `/commit`. Без демо незнакомый человек не поймёт ценность.

4. **README** — переписать с позиции нового пользователя: объяснить двухслойную структуру репо, показать что устанавливается через `template/`.

**Почему не Cookiecutter (отказ от ADR-003):** продукт — workflow-слой, пользователь приходит со своим кодом. Code scaffolding — другой инструмент, не цель этого репо.

**Последствия:**

- ADR-003 переходит в статус «Отклонено»
- Задача Cookiecutter удаляется из `to-do.md`
- Мейнтейнерский слой остаётся на русском (внутренний инструментарий)

## ADR-013: Управление жизненным циклом ассистента — опциональные шаги install.sh и скрипт uninstall.sh

**Статус:** Принято

**Проблема:** `install.sh` всегда создаёт коммит и ветку `dev` — это мешает при установке в существующий проект. Нет способа удалить ассистента из проекта без ручной зачистки файлов, `.gitignore` и `.git/info/exclude`.

**Решение:**

1. **install.sh — опциональные коммит и ветка dev**: два отдельных вопроса с дефолтом `n`, задаются после вопросов о видимости ассистента. Если коммит=n и в репо нет коммитов — вопрос о ветке пропускается (ветку нельзя создать в пустом репо).

2. **scripts/uninstall.sh** — новый скрипт, доступный через curl. Алгоритм:
   - Проверить наличие `.git` в текущей директории
   - Показать список файлов/директорий ассистента, которые будут удалены
   - Спросить отдельно про `.markdownlint.json`
   - Запросить подтверждение
   - Удалить: `.claude/`, `.context/`, `CLAUDE.md`, `WORKFLOW.md` (и `.markdownlint.json` при согласии)
   - Вычистить блок `# workflow-template` до конца файла из `.gitignore` (если существует)
   - Вычистить блок `# workflow-template` до конца файла из `.git/info/exclude` (если существует)
   - Сообщить о результате; коммит оставить на усмотрение пользователя

**Последствия:**

- Установка в существующий проект стала полностью управляемой — пользователь сам решает про коммит и ветку
- Появился путь выхода: ассистент удаляется без следов через один скрипт
- `SETUP.md` обновляется: добавить раздел «Удаление»

## ADR-012: Улучшение install.sh — корректная работа с .gitignore и управление видимостью ассистента

**Статус:** Принято

**Проблема:** `install.sh` перезаписывает `.gitignore` при установке в существующий проект, уничтожая пользовательские исключения. Предупреждение о перезаписи расплывчато — не перечисляет конкретные файлы. Нет возможности скрыть присутствие ассистента в репозитории или коммитах.

**Решение:**

1. **Явный список перезаписываемых файлов** в предупреждении — статически, по текущему содержимому `template/` корневого уровня: `CLAUDE.md`, `WORKFLOW.md`, `.markdownlint.json`, `.claude/`, `.context/`. `.gitignore` указывается отдельно с пометкой «дополняется».

2. **`.gitignore` дополняется, не перезаписывается**: содержимое существующего `.gitignore` сохраняется, строки из шаблона добавляются с маркером `# workflow-template`, дубликаты пропускаются.

3. **Вопрос A — скрыть файлы ассистента из репозитория**: при `y` — `CLAUDE.md`, `WORKFLOW.md`, `.claude/`, `.context/` записываются в `.git/info/exclude`, а не в `.gitignore`. Причина: `.gitignore` коммитится и раскрывает присутствие ассистента маркером и содержимым; `.git/info/exclude` — локальный, в историю не попадает.

4. **Вопрос B — скрыть ассистента из сообщений коммитов**: при `y` — в `.claude/settings.json` добавляется ключ, отключающий Co-Authored-By (точное имя ключа уточняется при реализации).

**Последствия:**

- `install.sh` становится безопасным для установки в существующие проекты
- Пользователь явно управляет видимостью ассистента в репо и истории коммитов
- Ничто в закоммиченных файлах не раскрывает выбор инструментария при режиме скрытия

## ADR-011: `.context/notes/` — личные заметки владельца, исключены из git

**Статус:** Принято

**Проблема:** Каталог `.context/notes/` использовался для временных заметок и патч-файлов, но никак не описан и ничем не защищён от случайного коммита.

**Решение:** Зафиксировать назначение: `.context/notes/` — личное рабочее пространство владельца. Файлы туда никогда не коммитятся.

- В root `.gitignore` — паттерн `.context/notes/*.md`
- В `template/.gitignore` (новый файл) — тот же паттерн; попадает в новый проект через `install.sh`
- В `template/.context/notes/` — создана пустая директория с `.gitkeep`
- В `template/WORKFLOW.md` — добавлена строка в таблицу документации проекта

**Последствия:** Существующие файлы в `.context/notes/` (`cc-commands-simplify-patch.md`, `markdown-conventions-patch.md`) удалены.

## ADR-010: Отказ от template-mini

**Статус:** Принято

**Проблема:** `template-mini` (один файл `CLAUDE.md` вместо полной структуры `.context/`) планировался как облегчённый вход для простых проектов. Драфт готов, но не реализован.

**Решение:** Отказаться от реализации. Поддерживать два параллельных шаблона — дополнительный мейнтейнерский груз без явного запроса. Если потребность появится в реальном проекте — вернуться к вопросу.

**Последствия:** Задача «Приоритет 1: template-mini» удалена из `to-do.md`. Драфт в `.context/discussions/` сохраняется как история.

## ADR-009: Предпубликационный аудит — исправления перед выкладыванием на GitHub

**Статус:** Принято

**Проблема:** Аудит проекта перед публикацией выявил шесть проблем: два блокера функциональности, две нестыковки в документации, один пропуск в `.gitignore`, одна Markdown-конвенция.

**Решения:**

1. `install.sh` — добавить `git checkout -b dev` после init commit. Шаблон документирует модель `main/dev/feature/*`, но установщик не создавал ветку `dev` → первый «начинаем реализацию» и «закрываем задачу» падали.
2. `cc-export-chat.md` — удалить из обоих слоёв (`skills/meta/`) и убрать все упоминания. Скилл ссылается на `export_chat_history.py`, которого нет в шаблоне и который ни разу не использовался.
3. `settings.local.json` — добавить в `.gitignore`. Файл создаётся CC автоматически; содержит локальные абсолютные пути — не должен попадать в репо.
4. `.gitignore` — дополнить `.claude/chat_history/` и `.env`.
5. `CONTRIBUTION.md` — исправить устаревшую фразу `"обсудим задачу"` → `"обсудим"`.
6. `template/WORKFLOW.md` — добавить `text` к code block «Ветки» (нарушение Markdown-конвенции).

## ADR-008: Удалить раздел «Управление контекстом» из CLAUDE.md обоих слоёв

**Статус:** Принято

**Проблема:** Инструкция «после завершения любого режима или скилла — упоминай текущий уровень контекста (`/context`)» невыполнима: `/context` — UI-команда, доступная только пользователю. Модель не может ни вызвать её, ни узнать результат. Раздел занимает контекст и создаёт ложное ожидание автоматической проверки.

**Решение:** Удалить раздел «Управление контекстом» из `CLAUDE.md` (мейнтейнерский слой) и `template/CLAUDE.md` (шаблонный слой).

**Последствия:**

- Пользователь сам следит за заполненностью контекста через `/context`.
- Если потребуется — пользователь явно просит проверить контекст, модель отвечает «выполни `/context` сам».

## ADR-007: Отказаться от init-project.sh в пользу install.sh

**Статус:** Принято

**Проблема:** Два способа установки создают путаницу. `init-project.sh` требует клонировать весь шаблон — это загрязняет git-историю нового проекта. `install.sh` (curl) решает ту же задачу чище и без лишних зависимостей.

**Решение:** `scripts/install.sh` — единственный поддерживаемый способ установки шаблона. `scripts/init-project.sh` удаляется.

**Связь с ADR-003:** задача "Удалить scripts/init-project.sh" из плана cookiecutter выполняется досрочно и независимо.

**Последствия:**

- Удалить `scripts/init-project.sh`
- Обновить `SETUP.md`: убрать «Способ 2: через клон», оставить только curl
- Проверить `CONTRIBUTION.md` на упоминания `init-project.sh`

## ADR-006: Переименовать docs/ → context/

**Статус:** Принято

**Проблема:** `docs/` читается как "документация проекта" (API docs, user guide), но содержит рабочие артефакты рабочего процесса: blueprint, plan, to-do, status, decisions. Название вводит в заблуждение новых участников.

**Решение:** Переименовать директорию `docs/` в `.context/` в обоих слоях — мейнтейнерском (корень) и шаблонном (`template/`).

Точка-префикс выбрана намеренно: `.context/` — служебная директория рабочего процесса, не артефакт проекта. Аналогично `.git/`, `.claude/` — инфраструктура, а не содержимое. При этом blueprint и decisions остаются доступны напрямую для тех, кто знает куда смотреть, и не мозолят глаза тем, кто пришёл за кодом.

**Последствия:**

- `docs/` → `.context/` в корне репо
- `template/docs/` → `template/.context/` в шаблонном слое
- Обновить все ссылки на `docs/` в: `CLAUDE.md`, `CONTRIBUTION.md`, `SETUP.md`, `.claude/index.md`, `.claude/skills/meta/` (5 скиллов), `template/CLAUDE.md`, `template/.claude/index.md`, `template/.claude/skills/meta/` (5 скиллов)

## ADR-005: Добавить install.sh для установки через curl

**Статус:** Принято

**Связь с ADR-003:** ADR-003 планировал заменить `init-project.sh` на cookiecutter для поддержки стек-специфичных шаблонов. `install.sh` покрывает основной сценарий (публичное репо, без стек-специфики) без дополнительных зависимостей. ADR-003 откладывается до появления реального стек-специфичного кейса.

**Проблема:** Текущий процесс развёртывания требует клонировать workflow-template целиком, что загрязняет git-историю нового проекта клоном шаблона.

**Решение:** Добавить `scripts/install.sh` — скрипт, который скачивается через curl и устанавливает шаблон в свежий `git init`-репозиторий.

Механизм скачивания — tar.gz архив через GitHub:

```bash
curl -fsSL "${REPO}/archive/refs/heads/main.tar.gz" \
  | tar xz --strip-components=2 "workflow-template-main/template"
```

`--strip-components=2` убирает `workflow-template-main/template/` — файлы шаблона ложатся прямо в текущую директорию.

Сценарий для пользователя:

```bash
mkdir my-project && cd my-project
git init
curl -fsSL https://raw.githubusercontent.com/<GITHUB_USER>/workflow-template/main/scripts/install.sh | bash
```

**Разграничение скриптов:**

- `init-project.sh` — для мейнтейнера: запускается из клона репо, предполагает наличие `template/` рядом
- `install.sh` — для пользователей: скачивается через curl, работает в пустом `git init`

**Ограничения:** только публичное репо, всегда берёт `main`.

**Последствия:**

- `scripts/install.sh` добавляется в репо
- `SETUP.md` обновляется: оба способа установки документируются рядом
- GitHub username захардкожен в `install.sh` (читается из самого URL скрипта при `curl | bash` — можно не дублировать)

## ADR-004: Управление ростом decisions.md через cc-architect-sync

**Статус:** Принято

**Проблема:** `decisions.md` накапливается без ограничений. Замещённые и устаревшие ADR занимают контекст CC и затрудняют навигацию.

**Решение:** Расширить скилл `cc-architect-sync` шагом аудита `decisions.md`. При синхронизации скилл проверяет ADR со статусом "Заменено" и предлагает перенести их в `docs/history/decisions/`. Отдельный скилл не создаётся.

**Отложено:** понятие "стабильный ADR" (настолько встроен в практику, что не требует обсуждения) — до появления реального кейса.

**Последствия:**

- В `cc-architect-sync.md` добавляется шаг: проверить ADR со статусом "Заменено", предложить архивирование.
- `docs/history/decisions/` используется как архив (файлы вида `decisions-YYYY.md` или по теме).
- Изменение вносится в оба слоя: `.claude/skills/meta/` и `template/.claude/skills/meta/`.

## ADR-003: Замена init-project.sh на cookiecutter

**Статус:** Отклонено — заменено ADR-014

**Проблема:** `scripts/init-project.sh` — bash-скрипт с ручным заполнением плейсхолдеров и деструктивными операциями (`rm -rf .git`). Не поддерживает stack-специфичную структуру проекта (pyproject.toml, src layout и т.д.).

**Решение:** Заменить скрипт на cookiecutter-шаблон. `template/` переструктурируется в директорию `{{cookiecutter.project_slug}}/`. Переменные — в `cookiecutter.json` (project_name, project_slug, stack, package_name). Логика инициализации — в `hooks/post_gen_project.py` (git init, uv sync, удаление файлов не того стека).

Первый поддерживаемый стек: **python-uv** (src layout, pyproject.toml, .python-version, uv.lock).

Запуск: `uvx cookiecutter path/to/workflow-template` — не требует предустановки cookiecutter.

**Отложено:** поддержка `generic` стека (только workflow-слой, без кода) — до появления второго реального стека.

**Последствия:**

- Плейсхолдеры `{PROJECT_NAME}` в `template/*.md` мигрируют в `{{cookiecutter.project_name}}`.
- `scripts/init-project.sh` и `scripts/` удаляются.
- `SETUP.md` обновляется под новый процесс.

## ADR-002: Перенос index.md из docs/ в .claude/

**Статус:** Принято

**Проблема:** `docs/index.md` — навигатор для Claude Code (что читать и когда), но лежит в `docs/` рядом с рабочими артефактами проекта. Это нарушает логику: `docs/` для документации проекта, `.claude/` для инструментария CC.

**Решение:** Переместить `docs/index.md` → `.claude/index.md`. В шаблонном слое соответственно `template/docs/index.md` → `template/.claude/index.md`.

**Последствия:**

- `.claude/` становится полным домом для CC-инструментария: настройки, скиллы, навигатор.
- Ссылки на `docs/index.md` в `CLAUDE.md` и `CLAUDE.template.md` обновляются на `.claude/index.md`.

## ADR-001: Разделение мейнтейнерского и шаблонного слоёв

**Статус:** Принято

**Проблема:** Репо одновременно является местом разработки шаблона и самим шаблоном. Файлы документации (`docs/status.md`, `docs/plan.md` и др.) использовались и как живые рабочие документы мейнтейнера, и как шаблонные заготовки с плейсхолдерами — это конфликт.

**Решение:** Разделить на два независимых слоя в одном репо:

- Мейнтейнерский слой — корень репо: `CLAUDE.md`, `CONTRIBUTION.md`, `SETUP.md`, `docs/`, `.claude/skills/meta/`, `scripts/`.
- Шаблонный слой — папка `template/`: `CLAUDE.md`, `WORKFLOW.md`, `template/docs/` (с плейсхолдерами), `template/.claude/skills/meta/`.

Развёртывание нового проекта — через скрипт `scripts/init-project.sh`: интерактивно заполняет плейсхолдеры, перемещает `template/` в корень, удаляет мейнтейнерский слой, переинициализирует git, привязывает новый remote, делает init commit.

Мейнтейнерская шпаргалка по работе с репо — `CONTRIBUTION.md` (не копируется в новый проект). Шпаргалка для нового проекта — `template/WORKFLOW.md`.

**Последствия:**

- `docs/` содержит реальную документацию мейнтейнера, скиллы (cc-status-report и др.) работают корректно.
- `.claude/skills/meta/` и `template/.claude/skills/meta/` — независимые копии, расходятся по мере развития.
- `CLAUDE.template.md` переименовывается в `template/CLAUDE.md`.
- `SETUP.md` остаётся в корне как инструкция по развёртыванию (удаляется скриптом).
